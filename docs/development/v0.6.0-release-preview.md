# v0.6.0 — LLM Removal: Release Preview

> **Status (2026-04-30): Preview — Not Yet Released.**
> Generated by `release_tracking next_release_preview` and refined for human readers. Use this document to drive the release ceremony for v0.6.0.

## TL;DR

DocuMCP v0.6.0 is a **breaking release** that removes the bundled multi-provider LLM client and aligns the server with MCP best practice: LLM execution belongs in the host, not the server. Hybrid semantic analysis is replaced by deterministic AST analysis; future LLM-driven features will use **MCP Sampling**.

**Driving ADR**: [ADR-014: Removal of Server-Side LLM Dependency in Favor of MCP Sampling](../adrs/adr-0014-removal-of-server-side-llm-dependency-in-favor-of-mcp-sampling.md). Supersedes Phase 2.5 of [ADR-009](../adrs/adr-0009-content-accuracy-validation-framework.md).

**Suggested version bump**: minor → `0.5.10` → `0.6.0` (semver minor for breaking changes pre-1.0; tagged `BREAKING CHANGE:` in CHANGELOG).

## Breaking Changes

### Removed APIs / tools

- **`simulate_execution` MCP tool** — gone. Hosts that called `tools/call simulate_execution` will get a "tool not found" error. There is no AST-only equivalent; remove the call from your host config.
- **`SemanticAnalyzer.useLLM` / `SemanticAnalyzer.confidenceThreshold` / `SemanticAnalyzer.llmConfig`** — gone. The `createSemanticAnalyzer()` factory now ignores these options (and warns on first call before deletion in v0.7.0).
- **`LLMClient` class** (`src/utils/llm-client.ts`) — gone. External orchestrators that imported `LLMClient` from `documcp` must vendor their own client or migrate to MCP Sampling.

### Removed environment variables

- `DOCUMCP_LLM_PROVIDER`
- `DOCUMCP_LLM_API_KEY`
- `DOCUMCP_LLM_MODEL`
- `DOCUMCP_LLM_BASE_URL`

These are silently ignored in v0.6.0 (with a one-time stderr deprecation warning) and removed entirely in v0.7.0.

### Removed documentation

- `docs/how-to/llm-integration.md` — deleted.

## Migration Guide

### If you used hybrid mode

You don't have to do anything for AST-only mode to keep working — that is now the default and only mode. If you specifically relied on LLM-grade semantic detection (e.g., your CI fails when an API doc is "about right" but an enum value changed semantically), one of two paths:

1. **Wait for MCP Sampling** — when host coverage broadens, DocuMCP will issue `sampling/createMessage` requests through the host, using the host's LLM and policies. No env vars; no separate cost stream.
2. **Move the LLM call to your orchestrator** — call your LLM provider yourself with the same prompt that DocuMCP previously used, then feed the result into `validate_content` as a hint.

### If you imported `LLMClient` programmatically

Vendor the file from the v0.5.x tag if you really need it; it will not be re-exported in v0.6.0. The recommended path is to migrate to MCP Sampling client-side.
